ME panel has **two parallel recurring-revenue $ families** (both by churn/CW/access date, USD), all built in the proc `sp_rebuild_me_bridge` (see [[me-bridge-deploy-arch]]):

**Family A — recognized (Anshul mart, LAGS ~2 months):** `RRA $`=`cw_lf_current_mth_rt_usd` (by CW date), `RRL $`=`churn_lf_current_mth_rt_usd` (by churn date), `NRRA`=RRA−RRL. Basis = `recurring_revenue` (recognized). **Transfers EXCLUDED**, ProFood included (rra_usd = sum of smb+ent+profood segments). Matches the global panel.

**Family B — gross (opp_base `monthly_license_fee` x fx, FRESH):** `RRX $`=`xrra_usd` (by access date), `RRLX $`=`xrrl_usd` (by churn date, excl ONLY pre-access churns), `NRRX`=RRX−RRLX. **Transfers INCLUDED** (Jad Jun 2026: a member moving in = access, NRRX nets the 10k->5k relocation delta). ME-panel-specific (Jad's Apr-2026 access-date ask), NOT in the global panel.

Key facts learned (Jun 2026 deep-dive):
- **`recurring_revenue` ≈ `monthly_license_fee`** (gross ≈ recognized, within ~10%). So "gross vs recognized" is NOT a big driver.
- **THE FX CURRENCY BUG (was making RRX ~3x RRA — the "way higher" Jad flagged):** `fx_by_country_month.month` was always NULL because `currency_exchange_rates.month` is a STRING TIMESTAMP like `'2026-02-01 00:00:00+00:00'`; the old `PARSE_DATE('%Y-%m-%d')` / `SAFE_CAST(... AS DATE)` returned NULL -> fx never joined -> `COALESCE(fx,1)` summed every opp_base $ (xrra/xrrl/nrrx) in **mixed LOCAL currency** (~3x high, Saudi/UAE fx≈0.27). **FIX:** `DATE_TRUNC(DATE(SAFE_CAST(cer.month AS TIMESTAMP)),MONTH)` + a `fx_latest` carry-forward CTE (rate table stops ~2026-02, so recent/future CW months fall back to latest rate). This was the entire inflation; NOT a basis difference. Family A (mart) was always correct USD.
- **RRX is a flow not a stock** — 100% of K-Delivery closed-wons access (0 pre-access churns in that population), so cumulatively RRX = RRA (same kitchens), monthly offset by the ~73-day (med ~31d) CW->access lag. So month-to-month RRX can be above/below RRA legitimately; "not way higher" = scale/cumulative, not per-month <=.
- **RRLX** is the user's rename of the old mislabeled "RRL $ (post-access) - excl pre-access churns". Pre-access churns are tiny (~2% of churns), so "post-access" was a near-zero distinction; the REAL difference vs RRL $ is gross/fresh/transfers-in (RRLX) vs recognized/lagging/transfers-out (RRL). Use RRL with RRA, RRLX with RRX (don't cross families).
- **CURRENT-MONTH LAG FILL:** recognized RRA/RRL load ~2mo late so the live month reads 0. Proc fills the live month only: `COALESCE(NULLIF(fm.rra_usd,0), IF(s.month_end <= LAST_DAY(CURRENT_DATE(),MONTH), raf.v, NULL))` where raf/rlf = fresh gross monthly_license_fee (excl transfers, to match A). Auto-reverts to recognized when the mart loads; future months NOT filled (they have churns but no CWs -> would show false-negative NRRA). Fill quality: RRA within ~5%, RRL ~15% low (provisional). The fill numbers are REAL Salesforce data (gross contractual LF of the live month's actual CWs/churns), just on the gross basis until recognized loads.

Bridge fx CTE also feeds nothing else important as a ratio (cw_duration weight cancels currency). bq month-parse gotcha applies anywhere `currency_exchange_rates.month` is cast. See [[me-panel-rrl-lf-only]].

**SOQL NULL-CHURN GOTCHA (Yazan reconciliation, Jul 9 2026):** Yazan flagged Discounted RR = 324K USD (June) vs the bridge's ME 2.73M. Root cause = his SOQL `WHERE ... AND Churn_Date__c > LAST_MONTH` — in SOQL (and SQL) `field > value` is FALSE for NULL, so it silently drops every never-churned (Churn_Date__c IS NULL) live customer. June decomposition on the SF mirror: future-churn-only (Yazan) = 268 opps / 325K; never-churned NULL (DROPPED) = 2,737 opps / 2.66M; already-churned (correctly excluded) = 7,307 opps / 8.4M. FIX for his query: `AND (Churn_Date__c = null OR Churn_Date__c > LAST_MONTH)` -> 3,005 live opps / 2.99M. Reconciliation walk 324K -> +NULL-churn 2.99M -> +accessed(occupied) 2.86M -> +Delivery-only 2.70M -> +dedup 2.70M ≈ bridge 2.73M (1.3% gap = fx CASE-map vs bridge dated currency table). CONFIRMS bridge Discounted RR is correct; Yazan's is the undercount. Also validated bridge FX converts to USD (independent CASE-fx bucket A = 324,778 ≈ his convertCurrency 324,402 to 0.1%). Definitional note surfaced: bridge Gross/Discounted RR use the OCCUPIED universe (accessed + Delivery + latest-CW-per-kitchen), matching the panel's Occupied Kitchens row + "same opps as Gross RR"; if Jad wants the broader "all closed-won live opps" (incl sold-not-yet-accessed + all products = S1 2.99M) that's a deliberate switch applied to BOTH gross+disc together (ratio unaffected).
