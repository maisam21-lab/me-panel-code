The ME panel (`me_panel_complete.gs` on Desktop) reads the Extract_K sheet (a BigQuery connected sheet mirroring `me_sales_panel_k_monthly`) by **fixed column POSITION**, not by header name. The `SRC` object maps `field -> column number`, and reads are `row[SRC.field - 1]`. So **inserting a column mid-list shifts every metric after it and silently mis-labels them** (numbers look plausible but are under the wrong names). Adding a metric SAFELY = **append at the very end** (no existing column moves), then verify `INFORMATION_SCHEMA ordinal_position == SRC value`.

**Column layout:** proc final SELECT outputs 1..N in order; `CREATE OR REPLACE TABLE AS SELECT` gives clean ordinals 1..N (the old "ae_deals ordinal 129" comment is STALE from the pre-self-contained bridge). Extract/`EXTRACT_HEADERS_EXPECTED` validates only cols 1-124; cols 125+ (ae_cw_prod_trial, ae_deals, and now the appended metrics) are read by SRC position but not header-validated — fine, just append past them.

**To add a metric end-to-end (verified pattern, Jun 2026):**
1. PROC (`me-panel-bq/setup/sp_rebuild_me_bridge.sql`): add a `..._monthly` CTE (per country + 'Middle East' UNION rollup), LEFT JOIN it in `joined`, and append the column LAST in the final SELECT (after the previous last col). Deploy + rebuild + verify the value.
2. PANEL (`me_panel_complete.gs`): add to `SRC` (= the new ordinal); add a block in `getWebBootBlockDefs_()`; add the `fromExtractME` read line in BOTH read functions; add to the FORMATS default, the null-default, the color map, and `GLOBAL_FULL_PANEL_ORDER`. (DATA array is only cols 1-124 — skip it for appended metrics.)
3. Verify `INFORMATION_SCHEMA.COLUMNS ordinal_position == SRC`. Then user: re-paste .gs -> refresh Extract -> Check Extract columns -> Build Panel_v2.
